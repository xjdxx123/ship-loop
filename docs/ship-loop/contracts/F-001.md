# Contract F-001 — cost accounting subcommand
Status: AGREED

## Done means
- `node scripts/ship-state.mjs cost --transcript <path>` exists as a new subcommand in `scripts/ship-state.mjs` (zero new dependencies) and prints exactly one JSON object to stdout — `{"input":N,"output":N,"cache_read":N,"cache_creation":N,"total":N}`, 2-space pretty-printed with trailing newline (same convention as `stats`), those five keys in that order, all values non-negative integers — then exits 0. Serialization is pinned to `JSON.stringify(obj, null, 2) + '\n'` with the result object built in that key insertion order; key order is therefore testable (ES2015 own-property order for non-index string keys is preserved by both `JSON.stringify` and `JSON.parse`), and the fixture test asserts the exact stdout string, which covers key order, values, formatting, and trailing newline in one assertion. In every exit-0 case stdout carries exactly this JSON and nothing else; all diagnostics go to stderr.
- Totals are cumulative sums over every transcript line that parses as JSON and carries a non-null `message.usage` object: `input` += `usage.input_tokens`, `output` += `usage.output_tokens`, `cache_read` += `usage.cache_read_input_tokens`, `cache_creation` += `usage.cache_creation_input_tokens`. A usage field contributes its value only when it is a non-negative integer (`Number.isInteger(v) && v >= 0`); anything else — absent, string, null, NaN, negative, fractional — contributes 0, which makes the "all values non-negative integers" output claim an enforced invariant rather than an assumption about transcript hygiene. `total` always equals `input + output + cache_read + cache_creation` (cache tokens are included because they dominate Claude Code session volume and the F-002 budget is a context/spend tripwire, not a billing invoice; lines are matched on the `message.usage` shape, not on `type === "assistant"`, to tolerate transcript-format drift).
- Tolerant parsing: blank lines, non-JSON lines, and valid-JSON lines without `message.usage` are skipped silently — they do not affect the totals, write nothing to stderr, and never change the exit code. (Silence is line-level only: the sole stderr outputs `cost` ever produces are the file-level unreadable warning and the missing-flag usage error below.)
- Fail-safe file handling: a missing or unreadable transcript prints the all-zeros object to stdout and exits 0 (the F-002 gate consumes this and must never crash a session; zeros = "no measured spend" fails safe by not triggering a false pause) — but not silently: it writes exactly one line to stderr, `ship-state: cost: cannot read transcript at <path>; reporting zeros` (where `<path>` is the `--transcript` value verbatim). Rationale: DESIGN_SPEC requires a one-line actionable message naming the item for missing/malformed input — a permanently wrong transcript path must not be indistinguishable from a healthy zero-spend run, while exit 0 + parseable stdout still guarantees F-002 never crashes. An empty or blank-only file is a legitimate "no spend yet" measurement, not a failure: all-zeros, exit 0, stderr empty. The only hard failure is invoking `cost` without a `--transcript` value (flag absent, or present with no value — `parseArgs` then yields boolean `true`): `fail()` with `cost requires --transcript` on stderr, exit 1, nothing on stdout. `--dir` is accepted and ignored (so the frozen feature-list verification command runs verbatim); `cost` never reads `feature_list.json` and works before/without `init`.
- A committed, byte-stable fixture `test/fixtures/transcript.jsonl` contains exactly these seven lines (JSON wrapping may carry extra realistic keys such as `type`, `message.id`, timestamps; the usage numbers are pinned):
  1. assistant line, full usage: input 100, output 200, cache_read 300, cache_creation 400
  2. assistant line, full usage: input 11, output 22, cache_read 33, cache_creation 44
  3. usage line missing both cache fields: input 5, output 7
  4. valid JSON line without `message.usage` (e.g. a user/summary entry) — contributes nothing
  5. malformed non-JSON line — skipped
  6. blank line — skipped
  7. usage line with junk fields: `input_tokens` is the string `"9"`, `cache_read_input_tokens` is `null`, `output_tokens` is 6 — contributes output 6 only (exercises the non-numeric rule in the committed fixture, and catches accidental `Number()` coercion: a coercing implementation reports input 125, not 116)

  EXPECTED TOTALS for this fixture are pinned: `{"input": 116, "output": 235, "cache_read": 333, "cache_creation": 444, "total": 1128}`. New `node:test` cases in `test/` (discovered by plain `node --test` from the repo root) assert: (a) the fixture run's stdout equals `JSON.stringify(expected, null, 2) + '\n'` exactly (string equality — pins key order, values, formatting) and stderr is empty; (b) missing file → all-zeros stdout, the exact one-line stderr warning, exit 0; (c) empty file → all-zeros stdout, empty stderr, exit 0; (d) both `cost` with no `--transcript` and `cost --transcript` (valueless) → exit 1, stdout empty, usage error on stderr.

## Out of scope

- Notifications on any threshold — F-004.

## Edge cases owned
- Missing transcript file → all-zeros JSON on stdout, exactly one warning line on stderr (`ship-state: cost: cannot read transcript at <path>; reporting zeros`), exit 0.
- Unreadable transcript file (permission denied, or path is a directory → EISDIR) → same as missing: zeros on stdout, the one-line stderr warning, exit 0 (any `readFileSync` failure takes this path).
- Empty file or file of only blank lines → all-zeros JSON, exit 0, stderr empty (a fresh transcript is a valid zero measurement, not a read failure).
- Malformed (non-JSON) lines interleaved with valid lines → skipped silently; valid lines still counted.
- Valid JSON lines without `message.usage` (user/system/summary entries, JSON scalars or arrays) → skipped silently.
- `usage` present but cache fields absent → cache fields contribute 0; `input_tokens`/`output_tokens` still counted.
- `usage` fields that are not non-negative integers (string, null, NaN, negative, fractional) → that field contributes 0; the line's other conforming fields still count.
- Missing `--transcript` flag, or `--transcript` with no value (`parseArgs` yields boolean `true`) → the only hard failure: `fail()` to stderr, exit 1.

## Evaluator response
Verdict: AGREED after five tightenings, all incorporated into the sections above. Scope is sound (cache tokens in `total` is right for a context tripwire; dedup overcounting deferred in the conservative direction is acceptable and documented), so no counter — but the proposal as written left the totals, the silence policy, and the regression claim under-pinned:

1. **Pinned the fixture's expected totals as exact numbers.** "Assert the exact expected totals" was unfalsifiable while the implementer could choose both fixture and totals freely. The fixture is now exactly seven lines with pinned per-line usage values, and the expected output is pinned: `{"input":116,"output":235,"cache_read":333,"cache_creation":444,"total":1128}`; verification command 3 now checks exact equality per key instead of `>0` smoke checks. Distinctive values (333/444) make field-misrouting visible; the string-`"9"` line makes silent `Number()` coercion fail the pinned total (125 ≠ 116).
2. **Kept key order — but made it testable by construction.** Key order on non-index string keys IS stable under `JSON.stringify`/`JSON.parse` (ES2015 insertion order), so the claim stays; the fixture test must assert exact stdout string equality against `JSON.stringify(expected, null, 2) + '\n'`, which pins order, values, formatting, and trailing newline in a single assertion rather than a hand-waved "same convention as stats".
3. **Broke the dangerous silence on unreadable transcripts.** All-zeros + exit 0 + empty stderr on a missing file would make a permanently wrong `--transcript` path indistinguishable from a healthy zero-spend run — the F-002 budget tripwire would silently never fire. DESIGN_SPEC ("one-line actionable message, never a stack trace"; alerts must name the item) demands better: missing/unreadable now emits exactly one stderr line naming the path, while stdout stays parseable zeros and exit stays 0 so F-002 still never crashes. Empty/blank-only files remain fully silent — they are legitimate measurements.
4. **Made "no behavior change to existing subcommands" verifiable.** All 13 pre-existing tests are now enumerated by name in "Done means" and must pass unmodified; "13 at time of writing" alone would let a regression hide behind a renamed or deleted test.
5. **Closed two spec gaps.** (a) Counting rule tightened from "non-numeric contributes 0" to "contributes only if a non-negative integer" — otherwise NaN/negative/fractional inputs could falsify the declared "all values non-negative integers" output invariant. (b) The valueless-flag form `cost --transcript` (which `parseArgs` turns into boolean `true`, not a path) is pinned as the same usage error as the missing flag, with a required test and verification command.

Verification command 2 remains the frozen feature-list command verbatim (`node scripts/ship-state.mjs cost --dir . --transcript test/fixtures/transcript.jsonl`), now with its exact expected output stated.
