# Contract F-008 â€” types without TS: `// @ts-check` + JSDoc, CI typecheck, zero runtime deps
Status: AGREED

## Scope
Per TECH_SPEC "Language" (binding): NO TypeScript migration, NO build step, NO loader â€”
the engine stays bare-Node `.mjs`; typescript is a CI-only tool, "never a runtime or
install dep". This feature adds `// @ts-check` + JSDoc annotations to
`scripts/ship-state.mjs` and `scripts/validate-repo.mjs`, a CI typecheck step, and
nothing else. Semantic behavior is FROZEN: `node --test` must report the same
35 tests / 34 pass / 1 skipped byte-for-byte, with zero edits to any test file.

## Two reproduced defects in the feature-list verification command (forced corrections)
The frozen feature verification reads `npx -y typescript@5 tsc --noEmit --allowJs
--checkJs --target es2022 --module nodenext scripts/ship-state.mjs
scripts/validate-repo.mjs` exits 0. As written it can NEVER exit 0; both defects were
reproduced before any code was touched:
1. **npx form.** `npx -y typescript@5 tsc ...` fails with `could not determine
   executable to run` (npm 10.9.4): the typescript package ships two bins (`tsc`,
   `tsserver`), neither named `typescript`, so npx cannot pick a default and `tsc`
   is just an argument. The package must be supplied via `-p typescript@5` with `tsc`
   as the command.
2. **Node builtin types.** With the npx form fixed, tsc exits 2 with TS2307 Ă—10
   (`Cannot find module 'node:fs' ...`) and TS2580 Ă—5 (`Cannot find name
   'process'/'Buffer'`): node's type declarations live in `@types/node`, which a bare
   typescript install does not include, and this repo has â€” and keeps â€” no
   package.json / node_modules to host it. Resolution: install `@types/node@22`
   (matching CI's `node-version: 22`) into the same throwaway npx prefix and point
   `--typeRoots` at it via `sh -c` + `command -v tsc` (the npx prefix's `.bin` is
   prepended to PATH, so `$(dirname "$(command -v tsc)")/../@types` is the prefix's
   `node_modules/@types`). This honors "prefer CLI flags over adding files": no
   tsconfig.json, no package.json, no committed `.d.ts` shims â€” hand-rolled node
   shims were rejected because vacuously-loose declarations would mask exactly the
   API drift the check exists to catch.

## The pinned invocation (byte-identical in CI and in verification)
```sh
npx -y -p typescript@5 -p @types/node@22 sh -c 'tsc --noEmit --allowJs --checkJs --noImplicitAny --target es2022 --module nodenext --typeRoots "$(dirname "$(command -v tsc)")/../@types" scripts/ship-state.mjs scripts/validate-repo.mjs'
```
All five original flags preserved; majors pinned (`typescript@5`, `@types/node@22`).
**Dial: `--noImplicitAny`, NOT full `--strict`** â€” one-line justification: default
checkJs exits 0 on the JSDoc-free baseline (reproduced â€” a gate that passes before the
work exists verifies nothing), while full `--strict` additionally forces TS18046
catch-clause casts and TS2322 null-handling rewrites INSIDE existing executable lines
(reproduced: 84 errors), breaking the comment-only rule below; `--noImplicitAny` is the
largest dial whose green is reachable by comment-only additions, and it machine-enforces
the @param bar below on every future function instead of trusting review. The
reproduced red under this dial is 75 errors; green means all 75 fixed by annotation.
`--strict` graduation is a future feature, not v1.

## Done means
- **`// @ts-check` is line 2** (directly under the shebang) of both
  `scripts/ship-state.mjs` and `scripts/validate-repo.mjs`, so editors check the files
  even without the CI flags; `git grep -l @ts-check scripts/` lists exactly these two
  files.
- **JSDoc coverage bar** (machine-enforced by `--noImplicitAny`): every top-level
  function declaration, every top-level `const`-assigned arrow, every nested helper
  arrow, and every method of the `cmds` dispatch object carries a typed `@param` for
  each parameter; every function whose return value callers consume carries `@returns`
  (expression-bodied arrows with incidental, discarded returns â€” e.g. an
  `errors.push` wrapper â€” document params only). `fail()` is pinned
  `@returns {never}` â€” load-bearing, not cosmetic: TS's never-call analysis is what
  narrows `if (!dir) fail(...)` postconditions so guarded flag uses typecheck without
  casts. Named typedefs are pinned: `Feature` (field-for-field mirror of
  `validateFeature`'s runtime checks, with `Status` and `FeatureType` literal unions),
  `FeatureDoc`, `Stats` (mirror of `computeStats`'s return), `Totals` (mirror of the
  five-key `cost` JSON â€” keys `input`/`output`/`cache_read`/`cache_creation`/`total`,
  deliberately NOT the four transcript usage field names, which must keep file-wide
  count exactly 1 each), `Lesson`, and `HookInput` (`cwd`/`transcript_path`, both
  optional â€” documents the Stop-hook stdin surface). Flag values out of `parseArgs`
  are typed `Record<string, *>` (honest: a valueless `--flag` yields `true`, so values
  are `string | true` consumed under each command's own declared contract); each `cmds`
  method then declares its real per-flag contract (e.g. `cost`'s
  `transcript?: string|boolean` with the `typeof` guard narrowing it).
- **Escape hatches pinned to zero**: `@ts-ignore`, `@ts-expect-error`, and
  `@ts-nocheck` each appear 0 times in both files. Any future exception must amend
  this contract with a one-line reason and a new pinned count.
- **Comment-only diff, mechanically verified**: the diff to both engine files consists
  ONLY of added comment lines (JSDoc blocks and `@type` lines above declarations) â€”
  zero lines removed, zero existing lines modified (`git diff` shows no `-` hunks in
  `scripts/`). This is what guarantees every frozen pin survives byte-exact:
  F-002/F-010/F-012 greps (four usage field names Ă—1, `timeout: 8000` Ă—1,
  `budget gate` Ă—2, `spawnSync(` Ă—1, `const READ_CHUNK = 65536;` Ă—1,
  `readFileSync(transcript` Ă—0, `async`/`await` tokens Ă—0 â€” comment text included, so
  annotations may never use those words â€” `cmds[cmd](flags);` Ă—1, both escalation row
  templates Ă—1, `ship-loop: gate stopped (stall)` Ă—1, `human intervention needed` Ă—0)
  and the stream test's source regexes (`^const READ_CHUNK = (\d+);$` and
  `function sumTranscript\(transcript\) \{` â€” the signature stays byte-exact; new
  comments inside that body must not contain `readFileSync(`).
- **CI step**, placed after "Unit tests" and before "Structural validation" (types are
  a deeper gate than structure but must not mask a behavioral failure as a type
  failure), named `Typecheck (JSDoc via tsc --noEmit; typescript is CI-only)`, whose
  `run:` is the pinned invocation above, byte-identical. No other ci.yml change.
- **Zero-deps proof**: no `package.json` exists anywhere in the repo after this
  feature â€” pinned ABSENT, not merely "no runtime dependencies": TECH_SPEC says
  typescript is "never a runtime or install dep", and even a devDependencies-only
  package.json would make it an install dep and invite a node_modules. Also pinned
  absent: `tsconfig.json`, `jsconfig.json`, any committed `.d.ts`, `node_modules`.
  (With input files given on the CLI, tsc ignores tsconfig.json anyway â€” flags are the
  only honest carrier.)
- **Semantic freeze**: `node --test` â†’ 35 tests, 34 pass, 1 skipped, 0 fail, zero test
  edits; `node scripts/validate-repo.mjs` green. Latent looseness that typing SURFACES
  (e.g. a valueless `--dir` flows `true` into `join()` and crashes with a bare
  TypeError instead of the house `fail()` message) is typed to current behavior,
  reported as a finding in the loop report, and NOT fixed here.

## Edge cases owned
- **npx cache/offline**: `-y` reuses the npx cache after first download; CI always has
  network. The two `-p` packages land in ONE prefix, so the `--typeRoots` derivation
  holds wherever npm puts it (path is machine-dependent, never committed).
- **`@types/node` minor drift**: major-pinned to 22 (matches `node-version: 22` in CI
  and the v22 dev machines). A future breaking @types minor that errors CI is fixed by
  pinning tighter in the one place the invocation lives (ci.yml + this contract).
- **The stream test's source regexes** are the strictest hidden coupling: JSDoc may sit
  ABOVE `function sumTranscript(transcript) {` (the regex anchors at the signature) and
  comment lines inside its body are safe as long as they avoid the literal
  `readFileSync(`; both are guaranteed by the comment-only rule plus pin sweep above.
- **JSDoc text is grep-visible**: annotations must not contain any pinned literal
  (usage field names, `budget gate`, `spawnSync(`, the words `async`/`await`, row
  templates). Verification 5 is the regression net for accidental mentions.
