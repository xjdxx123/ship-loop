# Contract F-006 — /ship:rollback: revert a delivered feature, reopen it, record the learning
Status: PROPOSED

## Done means
- `commands/ship-rollback.md` exists: frontmatter `description` + `argument-hint`
  (`"<F-id> <reason>"`), numbered imperative steps, `$CLAUDE_PLUGIN_ROOT` script paths,
  house voice (terse, operator-grade, no emoji). The command implements exactly this
  step list — mechanisms pinned, prose free:
  1. **Parse** — first token of `$ARGUMENTS` is the feature id, the rest is the
     rollback reason; missing either → ask and stop. The id must exist in
     `docs/ship-loop/feature_list.json`.
  2. **Preflight** — `git status --porcelain` must be empty; dirty tree → list the
     files, instruct stash or commit first, stop. Read `merge_strategy` from
     `docs/ship-loop/BUILD_CHARTER.md` (row absent = `merge`, matching conductor).
  3. **Locate, branching on `merge_strategy`** —
     - `merge`: `git log --merges --grep "<id>" --format="%H %s"`, FIRST match wins
       (newest; delivery merge subjects carry the id per conductor merge flow). Zero
       matches → report and stop; the command never guesses a hash.
     - `pr`: `gh pr view "ship/<id>" --json state,mergeCommit`. `OPEN` → `gh pr close`
       + delete the branch (origin + any local copy), skip the revert (nothing merged,
       nothing to revert — conductor step 6 semantics), continue at reopen. `MERGED` →
       the human already merged: take `mergeCommit.oid` as the delivery commit (empty →
       fall back to the `--merges --grep` search) and fall through to the merge-mode
       revert. No PR or `CLOSED` → report findings and stop.
  4. **Revert** — `git revert -m 1 <hash>` (mainline = first parent; conductor merges
     are always `--no-ff`). A squash/rebase-merged PR commit has one parent → plain
     `git revert <hash>` for that case only. On conflict, never auto-resolve: leave the
     revert in progress, list conflicted files (`git diff --name-only --diff-filter=U`),
     print the way out (resolve + `git revert --continue`, or `git revert --abort`)
     plus the exact reopen/learn commands to run by hand afterwards, stop.
  5. **Reopen** — `node "$CLAUDE_PLUGIN_ROOT/scripts/ship-state.mjs" set --dir "$PWD"
     --id <id> --status pending --passes false --note "rolled back <ISO-date>:
     <reason>"`. `--passes false` is load-bearing: `next` computes dependency
     eligibility from `passes`, so a rolled-back feature must stop satisfying its
     dependents.
  6. **Learn** — `{"lesson":"<id> rolled back: <reason>","tags":["rollback","<id>"]}`
     on stdin to `node "$CLAUDE_PLUGIN_ROOT/scripts/ship-state.mjs" learn --dir "$PWD"`;
     commit the state files.
  7. **Report** — what was reverted (hash + subject, or PR closed + branch deleted),
     what reopened (id → `pending`, attempts preserved), next action (`/ship:resume`).
- Diff touches ONLY `commands/ship-rollback.md` and this contract file. No engine
  change: `scripts/ship-state.mjs` already covers reopen (`set`) and learning (`learn`);
  no new status value, no schema or hook edits, no conductor edits (pr-mode rollback
  semantics were pinned there by F-005 — this command implements them, it does not
  re-document them).
- `node scripts/validate-repo.mjs` exits 0 (frontmatter present, no dangling
  cross-reference) and `node --test` stays 30 pass / 0 fail (1 pre-existing skip) with
  zero test edits.

## Out of scope
- Engine changes: `scripts/ship-state.mjs`, `hooks/gate.sh`, schema, tests. No
  `rolled_back` status — `pending` + note + learning is the frozen TECH_SPEC design.
- Conductor/skill edits: `skills/conductor/SKILL.md` step 6 already carries the pr-mode
  rollback note (F-005); byte-identical here.
- Reverting a revert / re-delivery mechanics — the reopened feature flows through the
  normal round protocol; nothing special on the second pass.
- Multi-feature rollback, commit ranges, rolling back the whole round.
- Conflict auto-resolution of any kind, including "trivial" ones.
- gh auth/setup; README and release docs (F-009).
- `scripts/validate-repo.mjs` changes — pure command-asset feature; behavioral
  verification is the grep list above.

## Edge cases owned
- Re-delivered feature (deliver → rollback → deliver → rollback): `git log` is
  reverse-chronological, so the FIRST `--merges --grep` match is the newest delivery;
  revert commits are non-merges and can never match `--merges`.
- Squash/rebase-merged PR: the delivery commit has one parent — `git revert -m 1`
  errors on non-merges, so the command drops `-m 1` for single-parent commits.
- Dirty tree: `git revert` would refuse or, worse, interleave in-flight work with the
  rollback — the command stops on ANY non-empty `git status --porcelain` before
  touching history.
- Conflicted revert: the in-progress revert is left in place WITH the exact reopen +
  learn commands printed, so a hand-finished rollback cannot strand the feature as
  `passed` in state.
- `set --status pending` alone leaves `passes: true` (the engine only flips `passes`
  when the flag is present) and dependents would still build on the reverted feature —
  step 5 pins `--passes false`.
- pr mode, PR merged by the human: `gh pr view --json mergeCommit` names the exact
  commit — no grep guessing; the `--merges --grep` fallback covers an empty API field.
- Wrong or never-delivered id: zero matches → stop with a report; the report always
  shows hash + subject before anything is reverted, so a same-prefix id collision
  (F-006 vs F-0060, possible past 999 features) is visible to the human.
- The reason lands in three greppable places — feature note, learnings.json, the state
  commit — so `lessons --grep rollback` and the retrospective both find it.
