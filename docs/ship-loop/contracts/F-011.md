# Contract F-011 — pr mode x depends_on: hold dependents until the dependency PR merges
Status: PROPOSED

## Problem being fixed
F-005's evaluator finding, verbatim scenario: in pr mode a feature is `passed` at PR
creation (`$SHIP set --passes true`), and the engine's `next()` gates dependents on
exactly that bit (`scripts/ship-state.mjs`: `passedIds` = features with `passes`,
eligible = pending whose `depends_on ⊆ passedIds`). So the moment the PR exists —
unmerged — `$SHIP next` starts offering the dependents. But Dispatch-pairs worktrees
are cut from main (`git worktree add ../wt-<id> -b ship/<id>`), and main does not
contain the unmerged PR's code: the dependent's implementer builds against a base that
lacks its declared dependency. In merge mode this is unreachable (passed = `--no-ff`
merged into main before the next Select); pr mode opened the gap because merging moved
to the human/CI. The conductor needs an explicit dependent-gating rule for the pr path.

## Design decision (made here, binding)
Three candidates:
- (a) **Stacked worktrees** — dependents branch from the dependency's `ship/<dep-id>`
  instead of main. Fastest (no waiting on the human), but PR merge order becomes
  load-bearing: the dependent's PR diff double-counts the dependency until the base PR
  merges, and a squash- or rebase-merged dependency PR (the default button in many
  repos) rewrites the commits the stack sits on — the dependent branch is orphaned,
  its PR conflicts on arrival. Keeping stacks honest needs base retargeting
  (`gh pr edit --base`) and ordering bookkeeping that must survive relay legs —
  machinery the conductor would carry as judgment prose, silently corruptible.
- (b) **Hold dependents** — the conductor additionally filters `$SHIP next` output in
  pr mode: a feature whose `depends_on` names a pr-passed-but-unmerged feature is not
  dispatched until `gh pr view ship/<dep-id> --json state` shows `MERGED`. Slower
  (dependent chains serialize on human merge latency) but trivially correct: every
  worktree still branches from main, and main provably contains every dependency at
  dispatch time. Zero rebase risk, zero new state (the check is a stateless `gh`
  query), and the starvation failure mode degrades into the loop's existing observable
  blockage path (Select-empty → NEEDS_HUMAN.md row → notify).
- (c) **Charter key, both configurable** — buys (a)'s speed only by accepting (a)'s
  corruption modes, doubles the doc surface, and widens the profile-merge matrix for a
  knob no run has asked for.

**Decision: (b), hold-by-default**, with a one-line pointer that advanced users may
stack manually (accepting the orphaned-stack risk); the loop itself never stacks. pr
mode exists precisely to put the human in the merge path — waiting on that human is
the mode's contract, not a defect. If dogfood later shows merge latency dominating, a
follow-up feature can add (c) on top of this rule without unpinning it.

**The engine is NOT modified.** `next()` gating on `passes: true` is correct and
frozen TECH_SPEC semantics ("the feature is `passed` when the PR is created"); the gap
is a workspace-base problem, and the conductor owns workspace creation. Patching
`next()` would also drag network/`gh` calls into the pure-state engine. The rule text
says this explicitly so nobody "fixes" `scripts/ship-state.mjs`.

## Pinned edits (exact text)

skills/conductor/SKILL.md — round step "Apply verdict", `pr` path: INSERT after the
line ending "(withheld-secret), never silently fall back to `merge`." and before the
"     Either path:" line:

```
       - Dependent gating: `passed` flips `passes: true`, so `$SHIP next` rightly
         starts offering this feature's dependents — but their worktrees branch from
         main (Dispatch pairs), and main lacks the unmerged PR. Hold them: at Select,
         filter the `$SHIP next` output — do NOT dispatch a feature whose `depends_on`
         names a pr-passed feature until, for each such dependency,
         `gh pr view ship/<dep-id> --json state` shows `MERGED`; anything else (OPEN,
         CLOSED, no `gh` answer) → treat the unmerged dependency as still-blocking.
         On `MERGED`, sync first (`git -C "$PRODUCT_DIR" pull --rebase`) so worktrees
         branch from a main that contains the dependency. The filter is conductor
         judgment — the engine's `next()` gates on `passes` by design; do not patch
         it. A hold that empties the round is Select's blockage case; the summary
         names the PRs awaiting merge. Advanced users may stack by hand (worktree off
         `ship/<dep-id>`), accepting that a squashed or rebased dependency PR orphans
         the stack; the loop never stacks.
```

## Out of scope

  (frozen TECH_SPEC: `passed` at PR creation); no schema/status change, no `gh` calls
  from the engine. The rule is conductor judgment applied to `$SHIP next` output.
- Stacked-PR automation: base retargeting (`gh pr edit --base`), stack ordering,

  the loop never performs it.
- Auto-merging dependency PRs (`gh pr merge`, auto-merge, branch protection): merging

## Edge cases owned
- **Stale local main after an origin-side merge**: in pr mode the human merges on
  origin while local main accrues the conductor's state commits; a worktree cut from
  stale local main would still lack the dependency even after `MERGED`. The rule pins
  `git -C "$PRODUCT_DIR" pull --rebase`: state commits (docs/ship-loop bookkeeping)
  rebase cleanly over the merged product code — disjoint paths by construction.
- **Dependency PR closed without merging**: CLOSED is not MERGED → still-blocking →
  dependents starve → Select-empty blockage row names the PR; the human's next action
  is reopen-and-merge or `/ship:rollback` the dependency (F-006). Fail-closed beats
  dispatching against a base that will never contain the code.
- **`gh` dies between PR creation and gating** (auth expiry, network): "no `gh`
  answer" is pinned into the still-blocking clause — the conductor never dispatches on
  an unverifiable merge state. (PR creation itself already parks when `gh` is absent,
  per the landed F-005 rule.)
- **Multiple pr-passed dependencies**: "for each such dependency" — every one must
  show MERGED; one unmerged PR holds the dependent.
- **PR merges while the loop is between rounds or relaying**: the check is stateless —
  first Select after the merge sees MERGED, syncs, dispatches. No bookkeeping to
  drift across relay legs (HANDOFF.md carries nothing new).
- **merge mode**: no feature is ever pr-passed, the predicate never matches, behavior
  is byte-identical to pre-F-011 — zero migration, same as F-005's row-absent rule.
- **Under-filled rounds**: filtering `$SHIP next --count` output can dispatch fewer
  than `max_parallel_pairs` while later-priority unheld features exist. Accepted:
  correctness over throughput; the conductor MAY re-run `$SHIP next` without `--count`
  and take the first unheld features, and the next round re-selects regardless.
- **Squash/rebase merge of the dependency**: the hold path is immune — the dependent's
  worktree starts from post-merge main, no stacked history to orphan. That immunity is
  exactly why (a) lost; the manual-stack pointer carries the warning for operators who
  stack anyway.
- **`gh pr view ship/<dep-id>` resolution**: the pr path pushes exactly `ship/<id>`
  (F-005 pinned), so head-branch lookup resolves even after origin branch deletion
  (the PR record keeps its head ref). If rollback + re-pass created two PRs from the
  same branch name, `gh pr view` prefers the open one — which is precisely the PR
  whose merge the dependent must wait for.
- **Dependent also depends on merge-passed features** (mixed `depends_on`): only
  pr-passed-and-unmerged entries hold it; `next()` already proved the rest passed.
