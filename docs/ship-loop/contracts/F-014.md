# Contract F-014 — valueless value-flags: table-driven usage errors at dispatch
Status: AGREED

## Problem being fixed
`parseArgs` in `scripts/ship-state.mjs` turns a trailing or valueless `--flag` into
boolean `true`, and ten value-requiring flags then leak that boolean into path/string
logic. Probed on this worktree (commit ccd416d):

- `validate --dir` (also `init/add/next/set/stats/gate/learn/lessons --dir`):
  `stateDir(true)` → `join(true, 'docs', 'ship-loop')` → raw
  `TypeError [ERR_INVALID_ARG_TYPE]` with a full Node stack trace, exit 1.
  DESIGN_SPEC "Must-have states" requires malformed input to get "a one-line
  actionable message, never a stack trace".
- `next --dir . --count`: `parseInt(true, 10)` → NaN → `eligible.slice(0, NaN)` →
  prints `[]`, exit 0 — a silent lie about eligibility (probed: exit 0, stdout `[]`).
- `set --dir D --id F-001 --passes` (valueless): `String(true) === 'true'` → silently
  sets `passes: true` — a state mutation indistinguishable from explicit
  `--passes true` (probed: exit 0, document flipped).
- `set ... --note` (valueless): appends the literal string `true` to notes (probed).
- `set --id` / `set ... --status` (valueless): `no feature true` / `bad status true` —
  one-liners, but nonsense assembled from the boolean leak.
- `lessons --dir . --grep` (valueless): `includes(true)` coerces to `'true'` →
  silently filters by the string "true", exit 0 (probed).

Fix: one table-driven guard at dispatch, mirroring the house pattern `cost` already
uses in-command (typeof guard + `fail()` one-liner), generalized so every
value-requiring flag is rejected BEFORE any command body — and therefore before any
state read or write — when its parsed value is boolean `true`.

## Done means
- **Table.** `scripts/ship-state.mjs` gains module-level `VALUE_FLAGS`
  (`@type {Record<string, string[]>}`), the per-command list of value-requiring
  flags, placed directly after `parseArgs` (pinned text below). Entries:
  `init: [dir, product]`, `validate: [dir]`, `add: [dir]`, `next: [dir, count]`,
  `set: [dir, id, status, passes, note]`, `stats: [dir]`, `gate: [dir]`,
  `learn: [dir]`, `lessons: [dir, grep]` — 16 (command, flag) pairs.
- **Helper.** One shared `usageError(cmd, flag)` (`@returns {never}`) emits the usage
  error via the house `fail()`: stderr gets exactly one line
  `ship-state: <cmd>: --<flag> requires a value`, exit 1, stdout empty, no stack
  trace. This is the file's single `requires a value` emission site (grep-pinned).
- **Dispatch check.** Exactly one new line between the existing usage-line check and
  the dispatch call:
  `for (const flag of VALUE_FLAGS[cmd] ?? []) if (flags[flag] === true) usageError(cmd, flag);`
  The guard fires only on boolean `true` (flag present, valueless); every string
  value — including the literal string `true` from `--passes true` — passes through
  untouched. The dispatch line `cmds[cmd](flags);` stays byte-identical (F-010 pin,
  count 1). Absent-flag messages are NOT this feature: `validate requires --dir`,
  `set requires --dir and --id`, `cost requires --transcript` etc. keep their exact
  current text and remain each command's own check.
- **Every flag classified** (full CLI surface):
  - value-required, table-guarded: `--dir` (init/validate/add/next/set/stats/gate/
    learn/lessons), `--product` (init), `--id` (set), `--status` (set), `--passes`
    (set — ruling below), `--note` (set), `--count` (next), `--grep` (lessons).
  - value-required, grandfathered in-command: `--transcript` (cost). Its typeof
    guard already rejects absent AND valueless alike with the frozen contract-F-001
    message `ship-state: cost requires --transcript`, pinned byte-exact by
    `test/ship-state.cost.test.mjs` ("cost without a --transcript value..."), which
    must pass byte-unmodified — listing it in the table would change that frozen
    stderr. The table comment names this exclusion.
  - no value to require: `--dir` (cost) stays accepted-and-ignored per F-001 (the
    frozen feature-list command `cost --dir . --transcript ...` runs verbatim);
    guarding a flag the command never reads adds failure for zero protection.
  - boolean-allowed: `--bump-attempts` (set) — valueless boolean `true` IS its valid
    form (`set --bump-attempts` bumps); a truthy string value also still bumps,
    unchanged.
  - exempt command: `stop-hook` takes no flags (reads hook JSON on stdin); it gets no
    table entry, so stray flags stay ignored — F-002/F-012 forbid the hook from
    failing or altering anyone's session.
- **Ruling: valueless `--passes` is a usage error** (not grandfathered). It mutates
  persistent state, and the silent `String(true) === 'true'` flip made `set --passes`
  indistinguishable from a deliberate `set --passes true`; a state-mutating flag must
  be explicit. Pinned stderr: `ship-state: set: --passes requires a value`. Every
  existing test calls `set --passes` with an explicit value and stays green
  unmodified.

  4. valued flags pass the guard: `set --passes true` then `--passes false` still

## Pinned edits (exact text)

scripts/ship-state.mjs — directly after the `parseArgs` function, INSERT:

```js
/**
 * F-014: per-command value-requiring flags, rejected at dispatch when
 * parseArgs yields boolean true (flag present, no value). Unguarded, that
 * shape either crashed join() with a raw TypeError (--dir and friends —
 * DESIGN_SPEC's bar is a one-line actionable message, never a stack trace)
 * or silently misread state (valueless `next --count` printed [], valueless
 * `set --passes` flipped a feature to passing). Deliberately NOT listed:
 * set --bump-attempts (boolean true IS its valid form), cost --transcript
 * (its own typeof guard already rejects absent and valueless alike with the
 * frozen contract-F-001 message), cost --dir (accepted and ignored per
 * F-001), and stop-hook (flagless; reads hook JSON on stdin and must never
 * fail a session — F-002/F-012).
 * @type {Record<string, string[]>}
 */
const VALUE_FLAGS = {
  init: ['dir', 'product'],
  validate: ['dir'],
  add: ['dir'],
  next: ['dir', 'count'],
  set: ['dir', 'id', 'status', 'passes', 'note'],
  stats: ['dir'],
  gate: ['dir'],
  learn: ['dir'],
  lessons: ['dir', 'grep'],
};

/**
 * One-line usage error for a value-requiring flag passed valueless (F-014).
 * @param {string} cmd @param {string} flag @returns {never}
 */
const usageError = (cmd, flag) => fail(`${cmd}: --${flag} requires a value`);
```

 * is boolean true — each command declares which value shapes it accepts, and

## Out of scope
- Unknown-flag rejection: `validate --bogus x` (or any flag a command never reads)
  stays silently ignored — a different, additive strictness feature; this contract
  guards only flags each command consumes.
- Value validation beyond presence: non-numeric `next --count x` (NaN slice → `[]`),
  unknown `set --status wish` (`bad status wish`), and a `--dir` pointing nowhere
  (readDoc's message) keep their current paths; only the boolean-true shape changes.
- The cost CLI surface: byte-frozen by F-001/F-002 (message, exit codes, --dir
  acceptance, fixture totals). This contract documents, and must not alter, it.
- parseArgs value-shape semantics: `--flag=value` syntax, repeated-flag last-wins,
  and the "next non-flag token is the value" rule are untouched.
- stop-hook behavior of any kind (F-002/F-012 own it; it stays guard-exempt).
- README/docs command references and shell callers (`hooks/gate.sh`,
  `scripts/relay.sh`, `scripts/headless.sh`) — no caller passes valueless flags.

## Edge cases owned
- Several guarded flags valueless at once (`set --dir --id`): parseArgs makes both
  boolean true; the guard reports the command's FIRST-listed flag — deterministic
  (table order is declaration order), pinned by test 6.
- Repeated flags: `validate --dir --dir .` ends as `dir: '.'` (pre-existing
  last-wins), so the guard sees the final post-parse shape and passes it — the guard
  judges what the command would actually receive.
- String `'true'` vs boolean `true`: `set --passes true` carries the STRING and is
  the documented explicit form (test 4); only the parser-produced boolean trips the
  guard.
- Usage errors cannot half-mutate: the guard runs before any command body, so before
  any readDoc/writeDoc/stdin read — proven by test 3's byte-identical document and
  by erroring in a dir with no feature list at all; `add`/`learn` exit before ever
  reading stdin.
- `gate --dir` valueless now exits 1 with the usage line on stderr and EMPTY stdout —
  distinguishable from gate's legitimate exit-1-with-stats-JSON while work remains.
- The frozen `next --dir . --count` verification runs against the repo root's real
  feature list, but the guard exits before any file I/O, so the command is
  deterministic in any cwd (test 2 runs it verbatim from ROOT).
- Hermeticity: every engine invocation in the new test file runs a tmpdir engine
  copy with a stub notify.sh sibling (house rule, F-002/F-012), so `node --test`
  can never pop a desktop notification even on failing runs; the two verbatim
  frozen-command assertions run the repo engine directly — they exit at the guard,
  before any state or notify path is reachable (same trade cost.test.mjs makes for
  its verbatim frozen command).
- The typecheck stays green with strictNullChecks off (the CI dial): `VALUE_FLAGS[cmd]
  ?? []` needs no cast, the helper is `@returns {never}` so TS's never-call analysis
  keeps post-guard narrowing intact, and no `@ts-ignore`/`@ts-expect-error`/
  `@ts-nocheck` appears (F-008 zero-pin holds).
