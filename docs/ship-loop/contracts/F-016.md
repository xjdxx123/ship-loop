# Contract F-016 — one-token hardening: -C "$PRODUCT_DIR" on the Dispatch-pairs worktree add
Status: AGREED

## Problem being fixed
`skills/conductor/SKILL.md` round step "Dispatch pairs" creates feature worktrees with
the only bare git command left in the file:

    create a worktree (`git worktree add ../wt-<id> -b ship/<id>`), mark

Every sibling git command already names the product repo explicitly — F-005's merge
(`git -C "$PRODUCT_DIR" merge --no-ff ship/<id>`) and push
(`git -C "$PRODUCT_DIR" push -u origin ship/<id>`), F-011's sync
(`git -C "$PRODUCT_DIR" pull --rebase`) — so the one command that CREATES the base
every other command assumes is also the only one that trusts the conductor's CWD.
`git worktree add` with no commit-ish cuts `-b ship/<id>` from the HEAD of whatever
repo (or linked worktree) contains the CWD, and resolves `../wt-<id>` against that
CWD. CWD drift therefore cuts the branch from the wrong HEAD and drops the worktree
under the wrong parent — partially undoing F-011's sync-before-cut, whose
`pull --rebase` explicitly refreshes `$PRODUCT_DIR`'s main while the very next
command may not be looking at it.

## Semantics confirmed (the -C reading F-011 needs)
`git -C <path>` runs git as if started in `<path>`; relative command-line paths then
resolve against `<path>`, not the invoking CWD. The pinned command is therefore
equivalent to `(cd "$PRODUCT_DIR" && git worktree add ../wt-<id> -b ship/<id>)`:
- the worktree lands at `$PRODUCT_DIR/../wt-<id>` — a sibling of the product repo,
  the file's existing placement convention — regardless of where the conductor is;
- `-b ship/<id>` cuts from `$PRODUCT_DIR`'s HEAD: exactly the main that F-011's
  `git -C "$PRODUCT_DIR" pull --rebase` just synced. Sync and cut now name the same
  anchor — which is the semantic F-011's sync-before-cut needs; its guarantee
  ("worktrees branch from a main that contains the dependency") previously held only
  while CWD == `$PRODUCT_DIR`.
Downstream, F-005's merge/push assume `ship/<id>` exists in `$PRODUCT_DIR`'s repo; a
drifted cut would create it elsewhere and the PASS path would fail at merge time.

## Edge cases owned
- **CWD == `$PRODUCT_DIR`** (the intended, common case): `-C` is a no-op — behavior
  byte-identical to pre-F-016, zero migration.
- **CWD in a foreign repo**: the bare form would cut `ship/<id>` from that repo's
  HEAD and drop `wt-<id>` beside the CWD; the implementer gets a worktree path that
  does not exist where the conductor reports it, and the eventual
  `git -C "$PRODUCT_DIR" merge --no-ff ship/<id>` fails (branch absent from the
  product repo). The `-C` form: right repo, right base, right placement.
- **CWD inside a linked worktree of the SAME repo** (e.g. a not-yet-removed
  `wt-<other>`): the nastiest drift — bare `worktree add` cuts from THAT worktree's
  HEAD (`ship/<other>`, a half-built sibling feature), silently embedding unreviewed
  code in the new branch while everything appears to work. `-C "$PRODUCT_DIR"` pins
  the cut to the main checkout's HEAD.
- **`$PRODUCT_DIR` unset/empty**: `git -C ""` leaves the working directory unchanged
  (git's documented empty-path semantics) — degrades to exactly the old behavior, no
  new failure mode; and a session with it unset has already broken every F-005/F-011
  sibling command anyway.
- **Relative-path reading**: with `-C`, `../wt-<id>` resolves against `$PRODUCT_DIR`
  (git chdirs before acting), NOT the invoking CWD — the worktree is always a sibling
  of the product repo, which is where every other mention of `wt-<id>` already
  assumes it lives.
