# Contract F-010 ‚ÄĒ streaming transcript reads (no full-file string)
Status: AGREED

## Problem being fixed
`sumTranscript` (the single shared transcript read site from F-002, called by both `cost` and the stop-hook) reads the whole file with `readFileSync(transcript, 'utf8')`. V8 caps strings at `0x1fffffe8` chars (536,870,888 ‚Čą 512MiB), so decoding a transcript past the cap throws `ERR_STRING_TOO_LONG`, the existing catch treats it as "unreadable", and the longest sessions ‚ÄĒ exactly the ones a budget tripwire exists for ‚ÄĒ report zero spend and evade the F-002 gate. Fail-open in the worst case. F-002's Out-of-scope explicitly delegated this to F-010.

## Decision: approach (b) ‚ÄĒ synchronous chunked reads, zero async plumbing
The dispatch site is synchronous today (`cmds[cmd](flags);` with handlers that `process.exit()` mid-flight), and `node:readline` is async-only. Of the two admissible designs:

- **(a) asyncify** `cost` + `stop-hook` and `await cmds[cmd](flags)` at dispatch ‚ÄĒ rejected. It rewrites the shared dispatch line for ALL nine commands to fix two, moves the stop-hook's multiple `process.exit()` points and its sequential best-effort breach writes into promise context (new unhandled-rejection and exit-ordering surface in the one code path whose silence rules are byte-frozen by F-002), and buys nothing functionally.
- **(b) CHOSEN: keep `sumTranscript` fully synchronous** ‚ÄĒ `openSync`/`readSync` in fixed `READ_CHUNK = 65536`-byte slices into one reused Buffer, `node:string_decoder`'s `StringDecoder('utf8')` to decode (a multi-byte UTF-8 character split across a chunk boundary decodes exactly as a whole-file read would), manual `'\n'` splitting with a carry string for the partial line, `decoder.end()` flushed into the final carry at EOF, `closeSync` in `finally` (best-effort). The entire diff is confined to `sumTranscript`'s body, its doc comment, the new constant, and the `node:fs`/`node:string_decoder` imports. Both call sites (`cost`, stop-hook) keep the exact call shape `const { totals, unreadable } = sumTranscript(t)`; the dispatch line and every other command are untouched ‚ÄĒ pinned by grep below (`cmds[cmd](flags);` count 1, zero `async`/`await` tokens file-wide). F-002 anticipated "node:readline" by name; this contract supersedes that wording with the readSync design for the dispatch-synchronicity reason above ‚ÄĒ the property F-002 actually relied on ("one-site fix inherited by both callers") holds identically.

Zero new dependencies: `node:string_decoder` is a builtin.

## Done means
- **No full-file string**: `sumTranscript` never materializes the whole transcript as one string or Buffer. Peak string cost is O(longest line + one 64KiB chunk): the carry holds at most the current line, each chunk decodes to ‚Č§64KiB of text, transient `split('\n')` parts are per-chunk. A transcript of any length whose every LINE fits in a V8 string sums to exact totals.
- **Per-LINE budget unchanged (out of scope, pinned degradation)**: a single line longer than the V8 string cap still cannot be represented; the carry concatenation then throws `RangeError`, is caught by the function's catch, and degrades to exactly the pre-F-010 whole-file shape ‚ÄĒ `unreadable: true` + all-zero totals (`cost` warns once on stderr; the hook stays silent). Counting such a line is beyond this feature.
- **Unreadable ‚áí all-zero totals, never partial sums**: ANY open or read error ‚ÄĒ at open, at first read (EISDIR), or mid-file ‚ÄĒ zeros the accumulated totals before returning `unreadable: true`. This matches the all-or-nothing semantics of the `readFileSync` it replaces; without the explicit zeroing, a mid-file I/O error would return a partial sum the old code never could.
- **Line semantics byte-identical to F-001/F-002**: same split on `'\n'` only (a `'\r\n'` file leaves the `'\r'` on the line and `JSON.parse` tolerates it, exactly as today); blank/whitespace lines skipped; non-JSON lines skipped silently; only non-null `message.usage` objects contribute; a usage field counts only when `Number.isInteger(v) && v >= 0`; file ending without a trailing newline still counts its final line (carry + `decoder.end()` flush); file ending with `'\n'` contributes no phantom empty line. `/dev/null` and empty files remain legitimate silent zero measurements (`readSync` returns 0 immediately).
- **All F-001 pins hold byte-exact**: fixture run prints the pinned `{"input":116,"output":235,"cache_read":333,"cache_creation":444,"total":1128}` via `JSON.stringify(obj, null, 2) + '\n'` (99 bytes, captured pre-change and `cmp`-verified post-change); missing/EISDIR transcript ‚Üí zeros + the exact one-line stderr warning, exit 0; `/dev/null` and empty ‚Üí zeros, empty stderr, exit 0; valueless/absent `--transcript` ‚Üí usage error exit 1. All 5 tests in `test/ship-state.cost.test.mjs` pass with ZERO edits.
- **All F-002 pins hold**: all 9 tests in `test/ship-state.budget.test.mjs` pass with ZERO edits (its hermetic engine-copy pattern copies the post-F-010 engine automatically); the F-002 grep pins stay exact ‚ÄĒ each of the four usage field names appears exactly once file-wide (so the new doc comment must not name any of them), `timeout: 8000` count 1, literal `budget gate` count 2 (the rewritten `sumTranscript` comment must not add a third). The stop-hook writes nothing to stderr in any case, unchanged.
- **All 13 tests of `test/ship-state.test.mjs` pass with ZERO edits** ‚ÄĒ 27 pre-existing tests total, none modified.
- **New `node:test` file `test/ship-state.stream.test.mjs`** (house style: spawnSync against the engine; tmpdir fixtures), covering:
  1. **Mechanism pin (red first)**: the engine source contains exactly one `const READ_CHUNK = 65536;`, and the extracted `sumTranscript` function body contains a bare `readSync(` call and NO `readFileSync(` ‚ÄĒ the transcript site provably streams. Red against the pre-F-010 engine (constant absent, `readFileSync` present), green after.
  2. **Chunk-boundary correctness, chunk strictly smaller than the lines**: a generated tmpdir fixture built as byte-exact Buffer segments sized from the source-extracted `READ_CHUNK` (= c), with in-test asserts that every planned offset landed: line A ~1.5c with a run of 3-byte UTF-8 chars positioned so the boundary at byte c falls 1 byte INSIDE one character ((c ‚ąí runStart) % 3 === 1 asserted) and its usage object after the boundary; line B long-ASCII with the boundary at 2c inside its padding before the usage object; a short CRLF-terminated usage line; a blank line; a non-JSON garbage line spanning the boundary at 3c and ending so its `'\n'` is exactly byte 4c (the first byte of a chunk ‚ÄĒ boundary-at-newline case, offset asserted); a junk-fields line (string `"9"`, null cache field ‚ÄĒ pins non-coercion through the new path); a final usage line with NO trailing newline (EOF flush). ## Out of scope
- A single LINE longer than the V8 string cap ‚ÄĒ degrades to the pinned pre-F-010 fail-open shape (zeros + `unreadable`), documented above; counting it would need a streaming JSON tokenizer nobody's transcript format justifies.
- Asyncifying any command or the dispatch site (rejected design (a) above).
- Any change to per-line counting rules, dedup of repeated `message.id` (F-001 declared), charter parsing, breach artifacts, or notify behavior (F-002 frozen).
- Non-UTF-8 transcript encodings, BOM stripping (both behave exactly as `readFileSync('utf8')` did).
- Memory-capping the carry for adversarial near-cap lines; tuning `READ_CHUNK` (65536 matches the fs-stream default highWaterMark; retuning requires revising this contract's grep pin and the boundary test sizes itself from the source constant, so it adapts).
- Running the >512MiB test in default CI (opt-in by env var, mandatory once in this feature's verification).

## Edge cases owned
- Transcript missing / EISDIR / permission denied ‚Üí `openSync` or first `readSync` throws ‚Üí zeros + `unreadable: true`; `cost` prints the exact F-001 warning; the hook stays silent. Byte-identical to today.
- Mid-file read error after partial accumulation ‚Üí totals explicitly zeroed; never a partial sum.
- Multi-byte UTF-8 character straddling a chunk boundary ‚Üí `StringDecoder` buffers the partial sequence; the line parses as if read whole (test 2 forces this with computed alignment).
- Line, number token, or garbage run straddling one or more boundaries; lines longer than a chunk ‚Üí carry logic reassembles exactly (test 2).
- `'\n'` landing exactly at a chunk edge ‚Üí empty first split part; no line lost or duplicated (test 2, offset-asserted).
- EOF without trailing newline ‚Üí final carry + `decoder.end()` flushed through the same line rules; EOF with trailing newline ‚Üí no phantom line.
- Truncated/invalid UTF-8 at EOF ‚Üí `decoder.end()` emits U+FFFD exactly as a whole-file decode would; the affected line fails `JSON.parse` and is skipped silently, as today.
- Empty file and `/dev/null` ‚Üí first `readSync` returns 0 ‚Üí silent zeros (tests + F-001 pins).
- `closeSync` failure ‚Üí swallowed; never affects totals, exit code, or output streams.
- Fixture and every existing caller behavior ‚Üí pinned by the 27 untouched pre-existing tests passing unmodified.
