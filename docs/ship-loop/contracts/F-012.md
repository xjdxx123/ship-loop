# Contract F-012 — stall escalation row: item, file, next action (+ gate-owned notify)
Status: AGREED

## Problem being fixed
The stall branch of `scripts/ship-state.mjs stop-hook` (spin count >= 3) appends
`... gate: loop stalled (N pending, state unchanged across 3 stop attempts) — human
intervention needed`. "human intervention needed" is the DESIGN_SPEC anti-example
("Something needs your attention!" without the item, the file, and the next action):
the row names the item but neither the file to inspect nor a concrete next action.
The budget row right above it (F-002) is the house shape — state change, file paths,
next action — and the stall row must match it. Additionally, the stall row is the one
NEEDS_HUMAN.md append with NO notification anywhere: the conductor convention
(skills/conductor Exit: "Notify on EVERY new NEEDS_HUMAN.md row at append time") cannot
cover it because a stall is precisely the conductor going absent, and TECH_SPEC's
notification decision exists for the walk-away case. A stalled unattended run that
never notifies is exactly the failure the notify feature exists for, so this fix also
wires the gate-owned notify call, mirroring F-002.

## Done means
- **New stall row, byte-exact.** The stall branch appends to `<cwd>/docs/ship-loop/NEEDS_HUMAN.md` exactly
  ``- [ ] ${ts} gate: loop stalled (${stats.pending} pending, state unchanged across 3 stop attempts) — run stopped; inspect docs/ship-loop/loop-run-log.md and docs/ship-loop/feature_list.json, then /ship:resume to re-enter the round or /ship:pause to stand down\n``
  where `ts` is `new Date().toISOString()` and `stats.pending` is the already-computed
  pending count. Item (loop stalled, N pending, unchanged across 3 stop attempts), files
  (loop-run-log.md to see what the loop was doing, feature_list.json to see what is
  stuck), next action (/ship:resume re-enters the conductor round per
  commands/ship-resume.md; /ship:pause writes PAUSED and stands the gate down per
  commands/ship-pause.md). The phrase `human intervention needed` disappears from the
  engine (grep-pinned to 0). The leading clause up to and including the parenthetical is
  byte-identical to today's row — the bug is the tail, and the diff stays minimal.
- **Stall semantics otherwise unchanged: stop allowed, spin cleared, NO PAUSED.**
  "run stopped" (not "run paused") is the truthful state-change phrase: the gate allows
  this stop and clears `.gate-spin`, but writes no PAUSED — the row's two offered
  actions are real alternatives, and a later stop in a still-stalled run legitimately
  re-arms the spin counter (3 more identical stops → one new row + one new notification
  per stall event, the same per-event dedupe philosophy as budget re-pause after PAUSED
  removal). Exit 0 with empty stdout and empty stderr on the stall path, as today.
- **Each stall step is individually best-effort (house rule from the budget branch).**
  The `appendFileSync` is now try/caught (today an EACCES crashes the hook with a stack
  trace — exactly what the stop-hook doc comment forbids); `unlinkSync` stays
  best-effort; notify is best-effort; `process.exit(0)` is always reached once the
  stall is decided. Stopping is the safe direction — a failed escalation write must not
  turn into a crashed hook in someone's session.
- **The stall fires exactly one notification, via a shared helper — machinery reused,
  not duplicated.** The F-002 notify block (sibling-resolved `notify.sh`, `existsSync`
  guard, `spawnSync('bash', [notifyPath, title, body], { stdio: 'ignore', timeout: 8000 })`,
  outer try/catch) moves verbatim into one module-level `notifyHuman(title, body)`
  helper called by BOTH the budget breach and the stall branch, so `timeout: 8000` and
  `spawnSync(` each still appear exactly once in the engine — the landed F-002/F-010
  grep pins (`grep -c 'timeout: 8000'` = 1, four usage field names once each,
  `budget gate` = 2) survive byte-exact instead of needing amendment. The budget call
  site keeps its position (after PAUSED is durable) and its byte-identical title/body;
  all 9 budget tests prove the refactor empirically, with ZERO edits to
  `test/ship-state.budget.test.mjs`.
- **Pinned notify strings (conductor convention: item, file, next action — never
  "needs attention").** Title `ship-loop: gate stopped (stall)` (parallel to
  `ship-loop: gate paused (budget)`: the gate stopped the run, category stall). Body
  ``NEEDS_HUMAN.md: loop stalled (${stats.pending} pending, unchanged across 3 stop attempts) — inspect docs/ship-loop/loop-run-log.md, then /ship:resume or /ship:pause``
  — the banner is single-line so it names the first file to open; the row carries the
  fuller two-file list. A missing, broken, or hanging notify.sh never changes the stall
  path's exit code, stdout, or the escalation row (notify fires after the append).
- **Stop-hook doc comment updated**: the stall bullet now describes append (item, file,
  next action), spin clear, notify-once, no-PAUSED re-arm semantics. The literal phrase
  `budget gate` still appears exactly twice file-wide (cost comment + budget bullet);
  the new text must not add a third.
- **Hermeticity of `test/ship-state.test.mjs` (forced by this feature, F-002 house
  rule).** The pre-existing stall test drives the engine to the stall branch, and that
  file's `BIN` is the repo engine whose notify.sh sibling is REAL — once the stall
  notifies, `node --test` would pop a desktop banner on every run, and under arbitrary
  gate bugs ANY stop-hook invocation in that file could (the stall branch, unlike
  budget, needs no `transcript_path`, so F-002's structural "cannot reach notify"
  argument no longer covers this file). Fix at the root: the file's module-level `BIN`
  becomes a tmpdir COPY of `scripts/ship-state.mjs` with an appending stub `notify.sh`
  beside it (budget-suite arrangement); sibling resolution makes the stub authoritative
  for every test in the file, even failing ones. The copy is made fresh from the repo
  file at module load, so the suite still tests the repo engine's exact bytes. The
  `gate.sh` end-to-end test still execs the REPO engine through `hooks/gate.sh` —
  acceptable at the F-002 standard: it constructs no state at all (no ACTIVE, no
  feature list), the weakest possible reach.
- **Sanctioned test-pin update, scoped to the single stall assertion.** F-012's frozen
  verification explicitly permits updating the stall test's pinned string. The
  `/stalled/` assertion in 'stop-hook blocks while features remain, allows after 3
  stale rounds with escalation' becomes a full-row regex anchored at both ends —
  `^- \[ \] <ISO-ts> gate: loop stalled \(1 pending, state unchanged across 3 stop
  attempts\) — run stopped; inspect ... \n$` — pinning the entire NEEDS_HUMAN.md
  content byte-exactly (timestamp shape aside) for that dir's 1-pending state. Exhaustive
  change inventory for `test/ship-state.test.mjs`, verifiable as exactly 3 diff hunks:
  (1) the `node:fs` import line gains `copyFileSync`; (2) the `const BIN = ...` line

  Every other line — all 13 `test(` declarations, every other assertion — byte-identical.
- **New `node:test` file `test/ship-state.stall.test.mjs`** (hermetic budget-suite
  style: module-level engine copy + appending stub notify.sh, plus a bare copy with no
  notify.sh sibling; no test in this file ever runs the repo engine), covering:
  1. byte-exact stall: an active run with 2 pending features (pins the

- **`skills/conductor/SKILL.md`** — its Exit text ("The unattended budget gate notifies

  conductor NEEDS_HUMAN appends (every new row ...)" states the every-row principle this
  contract implements for the one row the conductor cannot see.
- Stall-row dedupe across repeat stall events (one row + one notification per event is
  the budget path's re-pause philosophy, kept deliberately).
- PAUSED on stall, changed spin thresholds, spin-state redesign, or any change to the
  block-reason string — wording + notify only.
- Webhook/transport behavior, notify timeout rule (8000ms rule pinned by F-002, reused
  as-is via the shared helper).
- Surfacing stall/pause state in README or /ship:status (F-009).

## Edge cases owned
- Append fails (read-only state dir, EACCES) → caught; unlink and notify still
  attempted; exit 0 with empty stdout/stderr — the hook never crashes a session (today
  it does: uncaught throw, stack trace, exit 1; stall test 4 pins the fix).
- notify.sh missing beside the engine → `existsSync` guard skips notify; escalation row
  and exit code unaffected (stall test 3).
- notify.sh hanging or failing → 8s spawnSync cap + ignored stdio + try/catch via the
  shared helper; stall path silent on stdout/stderr regardless.
- `stats.pending` is 0 with features in_progress → row truthfully says "0 pending"
  (parenthetical reports pending exactly as today; `stats.done` was already false or
  the hook would have exited earlier).
- NEEDS_HUMAN.md already has rows (parked features, an earlier budget pause) →
  appended, never clobbered; the anchored single-row regex lives only in tests that
  construct fresh dirs.
- Stall then continued stalling: each new stall event (3 more identical stops) appends
  one row and fires one notification — bounded by the human-visible 3-stop cadence, and
  /ship:pause is the offered stand-down.
- Stall then progress: `.gate-spin` was cleared at escalation, so the next stop after
  real progress starts a fresh count (unchanged from today).
- Budget breach and stall in the same stop → budget wins (evaluated first, exits);
  unchanged ordering, budget suite pins it.
- Malformed stdin, no ACTIVE, PAUSED present, missing/invalid feature list, run done →
  unreachable for the stall branch via the untouched earlier checks.
