# ship-loop v0.2 — Iterate & Self-Evolution Design Spec

Date: 2026-06-10
Status: Approved (3 design iterations with owner)
Extends: 2026-06-10-ship-loop-design.md (v1 spec stays authoritative for everything not amended here)

## 1. Scope

Three additions, one schema-stable release (no ship-state.mjs changes):

1. **`/ship:iterate`** — re-entry path for products with ship-loop history: feedback-list
   intake, reproduce-first triage, delta questioning, document amendment + re-freeze.
2. **Self-evolution subsystem** — three-layer memory with a retrospective agent that
   distills run traces into project learnings (L1), a global user profile (L2,
   proposal-first autonomy), and playbook/protocol contributions (L3).
3. **`needsSpecialist` escalation** — implementers may request re-routing; only the
   conductor ever dispatches.

## 2. The three memory layers (global vs. project: both, layered)

| Layer | Travels with | Contents | Writer | Autonomy |
|---|---|---|---|---|
| **L1 project** | the repo (`docs/ship-loop/`) | learnings.json, contracts/, round history | state engine (existing) | automatic (existing) |
| **L2 user profile** | the human (`~/.claude/ship-loop/profile/`) | USER_PROFILE.md (stack/taste/workflow preferences), charter-defaults.json, intake-prefills.md, routing-overrides.md, proposals.md, evolution.log | retrospective agent | **proposal-first**: all writes start as proposals confirmed at next intake; low-risk classes may graduate to auto-apply; every applied change appended to evolution.log (append-only, auditable) |
| **L3 community** | the ecosystem | (a) protocol: plugin skills/agents — **human-PR only, never self-modified**; (b) **playbooks/**: provider execution knowledge in SKILL.md format | retro drafts locally; humans publish | drafts local → battle-verified → human-reviewed PR |

Distillation moves knowledge up: a fact seen in one project stays L1; a pattern seen in
≥2 products becomes an L2 proposal; a procedure that shipped ≥2 products becomes an L3
playbook draft. Conflict rule: **local overrides global** (a project's explicit choice
beats the profile default).

## 3. `/ship:iterate` (design-intake Path 2b)

Entry: `docs/ship-loop/` exists with frozen docs (regardless of who built the product).
Optional first-class input: a feedback/complaint list (file path or pasted text).

1. **Reproduce first**: triage agent attempts each complaint. Reproduced → bug feature
   whose `verification` IS the repro steps (fix-definition for free). Not reproduced →
   cannot-reproduce list back to the human; never enters the queue unconfirmed.
2. **Delta questions only**: read existing frozen docs; ask what changed / this round's
   success criteria / no-touch zones. Never re-ask answered questions. L2 prefills apply.
3. **Amend, don't rewrite**: PRD gains `## Round N amendments (date, motive)`; TECH_SPEC
   records change-decisions only; re-freeze. Prior contracts/ and passed features remain
   as regression floor and audit history.
4. Build loop unchanged. Delivery runs ship-deliver, then **retrospective (full)**.

Boundary line: **operate** maintains health (patrol + small fixes), **iterate** wages a
new campaign (systematic dissatisfaction → delta docs → full build loop), **ship** starts
from zero.

## 4. Retrospective agent

Triggers: after every ship/iterate delivery (full); after operate rounds (light).
Reads only existing traces: intake Q&A (each answer is a preference sample),
loop-run-log (rework rates, token actuals vs charter), contracts/ counter-patterns,
verdict rubric trends, NEEDS_HUMAN resolution patterns, capability-map hit rates
(specialist vs generic attempts).

Writes, by risk tier:

| Finding | Destination | Autonomy |
|---|---|---|
| Project-specific pitfall | L1 learnings.json | auto (existing `learn`) |
| Preference seen ≥2 products | L2 proposals.md | confirm at next intake → graduates to charter-defaults.json / intake-prefills.md |
| Routing win-rate pattern | L2 routing-overrides.md | proposal-first; may graduate to auto after a month of accepted proposals |
| Successful provider delivery ×2 | L3 playbook DRAFT (local) | human review before publish |
| Protocol improvement idea | stories/ draft in plugin repo | human PR only |

Every auto-applied change appends one line to evolution.log. No retro output may modify
plugin skills/agents/templates — structurally out of bounds.

## 5. Playbook layer (L3b) — the commercialization asset

Playbooks are provider-execution knowledge in **SKILL.md format** (zero new machinery:
capability-routing consumes them like any skill; they are the ecosystem's native sharing
unit, publishable to the repo's `playbooks/` and upstream to awesome-agent-skills).

Lifecycle: consume official skills (existing allowlist auto-install) → ship-deliver
executes + logs evidence → retro distills a draft after ≥2 successful deliveries on the
same provider → draft lives at `~/.claude/ship-loop/playbooks/<provider>/SKILL.md` →
battle-verified on the next delivery (first live use of a draft is flagged in the
handover) → human reviews → PR to plugin repo `playbooks/`.

Hard rules:
- **Provenance frontmatter required**: `derived-from` (run count), `providers`,
  `last-verified` (date), `success-rate`. The validator checks these fields on every
  published playbook.
- **No secrets, ever**: procedures and template variables only (`$VERCEL_TOKEN`), never
  values, account ids, or URLs containing tokens. Published artifacts must be secret-free
  by construction.
- **Routing priority**: official installed skill > verified local playbook > draft
  playbook (flagged) > generic implementer.

Compounding effect (why this matters commercially): every delivery makes the next
delivery cheaper; `playbooks/` becomes the accruing, community-shared moat.

## 6. `needsSpecialist`

Implementer exit JSON gains optional `needsSpecialist: "<category>"`. The conductor —
sole dispatcher, unchanged — re-routes the feature next round via capability-routing
(+ routing-overrides). Implementers still cannot dispatch agents (no Agent tool), which
preserves the maker/checker accountability boundary.

## 7. Out of scope (v0.2)

Multi-machine profile sync (git-backed profile repo — future), playbook marketplace,
auto-graduation tuning beyond the one-month rule, any ship-state.mjs schema change.
